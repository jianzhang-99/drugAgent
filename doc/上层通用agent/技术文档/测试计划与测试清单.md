# Drug-Agent 测试计划与测试清单

> 文档版本：v1.0
> 更新时间：2026-03-18
> 适用阶段：项目开发完成后的集中测试阶段
> 文档目标：指导 Drug-Agent 在测试阶段完成冒烟测试、功能测试、异常测试、回归测试与验收测试，并明确 AI 可辅助执行项与人工必测项

---

## 1. 文档背景

Drug-Agent 当前已基本完成核心开发，项目进入测试阶段。现阶段测试目标不再是验证“是否具备代码骨架”，而是验证系统是否已经达到“可稳定演示、可持续回归、可交付验收”的状态。

本项目当前包含两类重点能力：

- 上层通用 Agent 能力：统一对话入口、场景路由、工作流编排、SSE 流式输出
- 标书审查能力：任务创建、DOCX 解析、结构化字段提取、规则识别、风险结果输出

考虑到项目具备较多 AI 能力，本轮测试不仅要覆盖传统接口测试，还要明确哪些测试项可以由 AI 辅助执行，哪些测试项必须由人工判断。

---

## 2. 测试目标

本轮测试目标如下：

- 验证核心主链路是否全部可用
- 验证主要接口在正常输入和异常输入下表现稳定
- 验证标书审查链路对真实样本的输出是否合理
- 验证流式对话能力是否可稳定使用
- 验证已有单元测试和模块测试是否可作为回归基础
- 输出一份可执行、可复用的测试清单与测试方法说明

### 2.1 测试通过标准

满足以下条件，可以判定本轮测试基本通过：

- `P0`、`P1` 缺陷清零
- 核心接口均可稳定执行
- 标书样本可稳定完成创建任务与文档解析
- 风险识别结果能够输出可复核的依据
- SSE 链路无明显中断、错乱、无法结束问题
- 已有自动化测试能够稳定通过
- 至少完成 1 轮完整回归测试

## 3. 测试范围

### 3.1 本轮纳入范围

- 通用对话接口

- 多节点部署一致性
- 长时间稳定性 soak test

## 4. 测试阶段安排

### 4.1 Phase 1：冒烟测试

目标：确认项目具备进入详细测试的基础条件。

检查内容：

- 服务可启动
- Swagger 页面可访问
- 本地配置可正常加载
- 核心接口可调用
- 样本文件可上传
- `mvn test` 可执行

建议用时：0.5 到 1 天

### 4.2 Phase 2：功能测试

目标：验证需求功能是否按预期工作。

检查内容：

- 通用对话
- 场景路由
- SSE 流式输出
- 标书任务创建
- 文档解析
- 风险识别与结果输出

建议用时：2 到 3 天

### 4.3 Phase 3：异常与边界测试

目标：验证系统在异常输入和异常环境下是否仍然可控。

检查内容：

- 空参数
- 非法文件
- 非法 ID
- 重复请求
- 模型异常
- 空响应
- 非预期数据结构

建议用时：1 到 2 天

### 4.4 Phase 4：回归测试

目标：验证缺陷修复后未引入新的问题。

检查内容：

- 核心主链路回归
- 已修复缺陷专项回归
- 真实样本回归
- 自动化测试回归

建议用时：1 到 2 天

### 4.5 Phase 5：验收测试

目标：从业务演示与交付视角完成最终核验。

建议业务故事：

1. 用户发起监管问题咨询并得到回答
2. 用户创建标书审查任务
3. 用户解析指定文档并查看结构化结果
4. 系统识别围标风险并输出证据
5. 用户更换会话继续使用，系统不串话

建议用时：0.5 到 1 天

## 8. AI 可辅助测试项与推荐提示词

AI 在本项目测试阶段适合承担三类工作：

- 生成测试数据与测试问题
- 初步评估接口输出是否符合预期
- 对结构化结果进行一致性检查与缺陷初筛

但 AI 不适合独立做最终验收判断，尤其是在流式体验、页面表现、业务风险最终结论和误判边界上，仍需要人工复核。

### 8.1 通用对话测试提示词

#### 场景一：验证普通问答

用途：检查系统是否能稳定返回与药品监管相关的回答。

推荐提示词：

```text
你现在是一名测试助手。请围绕“药品监管、医疗耗材合规、招投标审查、合同预审”生成 20 条测试问题。

要求：
1. 问题长度分为短、中、长三类
2. 覆盖普通问答、追问、模糊提问、专业提问
3. 每条问题注明建议命中的场景
4. 输出为表格：编号、问题、建议场景、测试目的
```

#### 场景二：验证多轮记忆

用途：检查同一 `sessionId` 下的上下文保持能力。

推荐提示词：

```text
请为一个带会话记忆能力的 AI 系统设计 10 组多轮对话测试用例。

要求：
1. 每组包含 3 到 5 轮对话
2. 第 2 轮以后必须依赖前文信息才能正确回答
3. 覆盖人名、文件、病种、任务状态、规则解释等上下文
4. 输出为表格：用例编号、轮次、用户输入、预期系统行为
```

#### 场景三：验证场景路由

用途：检查问题是否会被正确导向标书审查、合同预审、风险预警或 fallback。

推荐提示词：

```text
请为一个多场景 AI Agent 生成 30 条路由测试输入。

目标场景包括：
1. TENDER_REVIEW
2. CONTRACT_PRECHECK
3. RISK_ALERT
4. UNKNOWN

要求：
1. 每条输入尽量自然，不要太像关键词堆砌
2. 同时包含容易误判的边界表达
3. 输出字段：输入文本、预期场景、为什么这样判断
```

### 8.2 文档解析结果初审提示词

用途：让 AI 对解析后的 JSON 结果做初步一致性检查，帮助发现漏字段、错字段、章节异常等问题。

推荐提示词：

```text
你是一名文档解析测试审查员。下面我会给你一份 DOCX 解析后的结构化结果，请你从测试角度帮我做初步检查。

请重点检查：
1. sectionTree 是否存在明显层级错误
2. paragraphBlocks / tableBlocks 是否存在空块过多问题
3. fields 中的电话、邮箱、报价、人员字段是否格式合理
4. extractionMeta 是否完整
5. 是否存在明显漏提取、重复提取、错位提取的迹象

输出格式：
1. 总体判断
2. 可疑问题列表
3. 建议人工重点复核的区域
```

### 8.3 风险识别结果初审提示词

用途：让 AI 协助对风险输出进行初筛，但不代替最终业务判断。

推荐提示词：

```text
你是一名围标风险测试评审助手。下面我会给你一份系统输出的风险识别结果，以及对应的部分原文证据。

请从测试角度判断：
1. 风险结论是否和证据基本一致
2. 是否存在“结论过强但证据不足”的情况
3. 是否存在“证据很多但结论过弱”的情况
4. 哪些部分应由人工重点复核

请不要直接替代人工做最终裁定，而是输出：
1. 初步一致性判断
2. 证据充分性评估
3. 人工复核建议
```

### 8.4 异常测试用例生成提示词

用途：让 AI 帮助补齐边界输入，而不是人工逐条想。

推荐提示词：

```text
请为一个 Spring Boot 后端接口系统生成异常测试与边界测试用例。

系统特点：
1. 有普通 JSON 接口
2. 有 SSE 流式接口
3. 有 multipart 文件上传接口
4. 有文档解析与 AI 风险识别能力

请输出：
1. 参数异常类用例
2. 文件异常类用例
3. 会话异常类用例
4. 结果一致性异常类用例

每条用例包含：编号、输入、异常点、预期行为、优先级
```

### 8.5 回归测试清单生成提示词

用途：在每轮修复后，让 AI 帮你快速整理“最小回归包”。

推荐提示词：

```text
请根据以下缺陷列表，帮我生成一份最小回归测试清单。

要求：
1. 优先覆盖最可能受影响的主链路
2. 标出每条回归项对应的缺陷编号
3. 输出为表格：回归项、覆盖模块、回归原因、执行方式、优先级
```

## 9. 人工必测项清单

以下内容不建议完全交给 AI 判定，必须人工介入：

### 9.1 交互与体验类

- Swagger 页面是否可正常访问
- SSE 是否有明显卡顿、丢片、重复片段、无法结束
- 前端工作台的任务创建、对话展示、错误提示是否清晰
- 大文本输出时的可读性和可用性

### 9.2 业务准确性终审

- 标书章节树是否真正符合原文结构
- 关键字段提取是否准确映射到了正确位置
- 围标风险结论是否满足业务上“可解释、可复核”的要求
- 合规样本是否被误判为高风险
- 灰度样本是否被简单粗暴地下结论

### 9.3 异常与鲁棒性

- 损坏 DOCX 是否导致系统崩溃
- 文件上传过程中断时系统表现
- SSE 中途断开后的后端表现
- 真实模型波动时系统是否仍可返回可接受结果

### 9.4 验收与演示

- 业务故事是否完整跑通
- 输出是否适合向老师、评审、客户做演示
- 最终结果是否能被非开发人员理解

## 10. 建议的测试执行顺序

建议按以下顺序推进：

1. 先跑自动化测试，确认代码基线稳定
2. 再做冒烟测试，确认环境、接口、样本链路可用
3. 再做主链路功能测试，优先覆盖通用对话与标书审查
4. 再做异常测试和边界测试
5. 再做真实样本与灰度样本测试
6. 缺陷修复后执行最小回归包
7. 最后执行人工验收测试

---

## 11. 缺陷分级建议

| 级别 | 定义 | 示例 |
|---|---|---|
| P0 | 阻断级问题 | 服务起不来、主链路不可用、核心接口报错 |
| P1 | 高优先级问题 | 场景路由错误、SSE 不稳定、风险结果明显错误 |
| P2 | 中优先级问题 | 局部功能异常、边界处理不佳、误报偏高 |
| P3 | 低优先级问题 | 文案、提示信息、日志细节问题 |

## 13. 当前阶段的落地建议

针对当前项目现状，建议你这样执行：

- 先把这份文档作为测试总控文档使用
- 每完成一个阶段，就把结果沉淀到缺陷清单和回归记录
- AI 主要负责生成问题、补充边界用例、辅助初审输出
- 人工重点负责主链路验证、流式体验、真实样本判断和最终验收

这样做的好处是：

- 既能发挥 AI 提升测试效率的优势
- 又不会把关键业务判断错误地完全交给 AI
- 方便你在答辩、汇报或交付时说明“测试工作是如何系统开展的”
