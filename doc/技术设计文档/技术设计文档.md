# Drug-Agent 药品智能监管系统 — 技术设计文档

> **版本**：v1.1 | **日期**：2026-02-27 | **技术栈**：Spring Boot 4.0 + 千问(DashScope) + MySQL + RAG + Vue 3

---

## 一、项目简介

基于大语言模型（千问/Qwen）构建一个药品智能监管系统，核心解决两个问题：
1. **药品用量异常难以发现** → AI 分析历史数据，自动识别异常并预警
2. **合规审查依赖人工** → 上传文件后与 AI 对话，自动比对法规判断合规性

---

## 二、核心概念澄清：我们不训练模型

| 方式 | 描述 | 本项目 |
|------|------|:---:|
| 模型训练/微调 | 修改模型参数让它学会新能力，需要 GPU、数据标注，成本高 | ❌ |
| **Prompt 上下文注入** | 每次调用时把数据塞到提示词里，让模型现场分析 | ✅ |
| **RAG 检索增强** | 先从知识库检索相关内容，再喂给模型回答 | ✅ |

**通俗理解**：千问模型就像一个请来的"专家顾问"，我们不需要教他学医药知识（他已经会了），我们只需要每次把"材料"（数据/文件/法规）递给他看，然后问他问题。

### 功能A：药品数据监管分析（详细拆解）

#### A1. Excel 批量导入

```
用户上传 Excel 文件
      │
      ▼
校验格式（列名是否正确）
      │
      ▼
逐行解析，数据清洗
      │
      ▼
批量写入 MySQL（drug_usage_record 表）
      │
      ▼
返回导入结果（成功N条，失败M条，失败原因）
```

**Excel 模板格式**：

| 药品名称 | 药品编码 | 使用量 | 单位 | 使用日期 | 科室 | 药品分类 | 操作人 |
|---------|---------|-------|------|---------|------|---------|-------|
| 阿莫西林胶囊 | YP001 | 200 | 盒 | 2026-01-15 | 内科 | 抗生素 | 张三 |

#### A2. 药品数据查询

- 支持按药品名称、日期范围、科室、药品分类等条件筛选
- 分页展示
- 前端表格 + 简单图表（折线图展示趋势）

#### A3. AI 智能分析（核心）

**这一步是怎么把数据"喂"给模型的：**

```
步骤1: 用户选择分析条件
   ↓    （如：分析"阿莫西林" 2026年1月-2月的用量）
   
步骤2: 系统从 MySQL 查询并聚合数据
   ↓    SELECT drug_name, usage_date, SUM(usage_amount)
   ↓    FROM drug_usage_record
   ↓    WHERE drug_name='阿莫西林' AND usage_date BETWEEN ...
   ↓    GROUP BY usage_date
   ↓    
   ↓    得到：每日聚合数据（如30天 → 30条汇总记录）

步骤3: Java 侧做统计计算
   ↓    · 日均用量: 150盒
   ↓    · 最大单日用量: 380盒 (1月20日)
   ↓    · 标准差: 45
   ↓    · 环比增长: +12%
   ↓    · 异常点: 1月20日用量超出2倍标准差

步骤4: 把统计结果组装进 Prompt
   ↓    "以下是阿莫西林近2个月的使用统计数据:
   ↓     日均用量150盒, 最大单日380盒(1月20日)...
   ↓     检测到异常: 1月20日用量异常偏高...
   ↓     请分析原因并给出监管建议"

步骤5: 调用千问模型 → 返回分析报告
       · 趋势总结
       · 异常原因推测
       · 风险等级：HIGH
       · 监管建议：建议核查1月20日...
```

> **关键点**：50万条原始数据 → SQL聚合为几十条统计摘要 → 塞进Prompt  
> 模型只需要"**看**"精炼后的统计数据来做推理分析，不是看50万条原始数据。

#### A4. 分析报告

- AI 返回的分析结果格式化展示
- 包含：趋势总结、异常清单、风险等级标签、建议列表
- 支持导出为 PDF（可选）

### 功能B：文件上传 + 合规对话（详细拆解）

#### B1. 文件上传

- 支持格式：PDF / Excel / Word / 图片
- 文件保存到本地 `uploads/` 目录
- 记录到数据库 `compliance_file` 表

#### B2. 文件解析

```
上传的文件
   │
   ├── PDF  → PDFBox 提取文字
   ├── Word → Apache POI 提取文字
   ├── Excel → Apache POI 读取行列 → 转为描述文本
   └── 图片 → 千问VL多模态模型 OCR识别
   
   ▼
提取出的纯文本 → 存储到数据库
```

#### B3. 合规对话（核心）

```
用户提问："这个文件里的药品采购流程是否符合《药品管理法》？"
   │
   ▼
① 从文件解析内容中截取相关片段
   │
   ▼
② 从法规知识库(RAG)检索《药品管理法》相关条款
   │
   ▼
③ 组装 Prompt:
   "参考文件内容: [文件片段]
    相关法规: [法规条款]
    用户问题: 采购流程是否合规？
    请给出判断和依据"
   │
   ▼
④ 调用千问模型 → 返回合规判断 + 法规引用
```

#### B4. 对话历史

- 同一文件可多次对话，上下文连贯
- 历史消息存 `chat_message` 表，下次可继续

---

### 功能C：法规知识库 RAG（详细拆解）

#### C1~C2. 法规文件入库流程

```
上传法规文件(如《药品管理法》PDF)
   │
   ▼
解析为纯文本
   │
   ▼
文本切片(每片约500字, 相邻片段有100字重叠)
   │
   例: 原文3000字 → 切为8个片段
   │
   ▼
每个片段调用千问 Embedding API → 得到向量(1024维浮点数组)
   │
   ▼
向量 + 原文 + 元信息 → 存入数据库(knowledge_chunk 表)
```

#### C3. 对话时自动检索

```
用户问题: "药品储存温度有什么要求？"
   │
   ▼
问题文本 → 调用 Embedding API → 得到问题向量
   │
   ▼
在知识库中计算余弦相似度, 取 Top-5 最相关片段
   │
   ▼
把这5个片段作为"参考法规"注入到 Prompt 中
   │
   ▼
模型基于这些真实法规内容回答(而不是凭"记忆"回答)
  → 避免模型编造不存在的法规条文
```

#### C4. 知识库管理

- 查看已入库文件列表
- 删除过期/错误的法规文件
- 测试检索效果（输入关键词查看返回的片段）

## 五、数据库设计

### 核心表

**1. drug_usage_record（药品使用记录）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| drug_name | VARCHAR(200) | 药品名称 |
| drug_code | VARCHAR(50) | 药品编码 |
| usage_amount | DECIMAL(10,2) | 使用量 |
| usage_unit | VARCHAR(20) | 单位(盒/支/瓶) |
| usage_date | DATE | 使用日期 |
| department | VARCHAR(100) | 使用科室 |
| drug_category | VARCHAR(50) | 药品分类(抗生素/麻醉药品等) |
| operator | VARCHAR(50) | 操作人 |
| created_at | DATETIME | 入库时间 |

**2. compliance_file（合规文件）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| file_name | VARCHAR(255) | 文件名 |
| file_path | VARCHAR(500) | 存储路径 |
| file_type | VARCHAR(20) | 类型(pdf/xlsx/docx) |
| parsed_content | LONGTEXT | 解析后文本 |
| created_at | DATETIME | 上传时间 |

**3. knowledge_chunk（知识库片段）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| file_id | BIGINT FK | 来源文件ID |
| chunk_index | INT | 片段序号 |
| content | TEXT | 文本内容 |
| embedding | BLOB | 向量(序列化存储) |
| source | VARCHAR(255) | 来源描述 |
| created_at | DATETIME | 创建时间 |

**4. chat_session + chat_message（对话）**

| 表 | 关键字段 |
|----|---------|
| chat_session | session_id, file_id, session_type(monitor/compliance), created_at |
| chat_message | session_id(FK), role(user/assistant), content, created_at |

## 六、关键技术难点

| 难点 | 解法 |
|------|------|
| 50万条数据无法直接给模型 | SQL聚合→Java统计→摘要注入Prompt，模型只看精炼数据 |
| 模型可能编造法规条文 | RAG检索真实法规注入Prompt，并要求标注条款来源 |
| 大文件超出Token限制 | 文本切片(Chunking)，每次只检索最相关的片段 |
| 多轮对话上下文丢失 | 服务端维护会话消息列表，每次调用带上历史消息 |
