# 医疗监管AI智能体 - 场景一：超长标书文件高度雷同与语义查重技术方案

## 1. 业务背景与挑战

在医疗监管与合规审查中，招投标环节的违规操作（如围标、串标、标书抄袭）是重点审查对象。传统基于关键字或简单文本比对的查重工具，难以应对标书这种极长（通常几百页）、排版复杂（包含大量表格、图片、红头格式）以及为掩人耳目进行过“洗稿”修改的非结构化文档。

**核心痛点：**
- **长文本超出大模型上下文窗口**：无法将完整的两份数百页标书直接全量提交给大模型比对。
- **排版格式复杂**：PDF/Word格式中的表格、段落直接转制文本极易丢失语义关联关系。
- **误判率高（假阳性）**：合法引用相同的国家法规、医疗行业标准时，传统查重工具易误判为“抄袭”，必须依赖大模型的“语义理解”与“法务逻辑”进行排除。

## 2. 总体系统架构设计

基于当前项目的底座 `Spring Boot 3.4.4` + `Spring AI Alibaba (1.0.0-M6.1)` 以及千问大模型底座，本场景采用成熟的 **高级 RAG (Retrieval-Augmented Generation) 架构 + 混合检索与重排 + 大模型裁决** 机制。

### 2.1 核心工作流
整体流程将高度自动化，分为文档工程、底库构建、双路核对四个阶段：

1. **文档工程（Ingestion Pipeline）**
   标书 A（待查本）与标书 B（基准本）分别经过多模态文档智能解析器，将排版复杂的 PDF 抽取为层次分明的 Markdown，完整保留各章节 `#` 标题层级和表格结构。
2. **切分与入库（Chunking & Embedding）**
   通过语义边界切分器（Semantic Chunking）将标书 B 切分为多个 500-800 Token 的文本块（Chunk）。通过 Spring AI Alibaba 接入百炼 `text-embedding-v3` 模型转化为高维向量，携带着章节、页码等元数据（Metadata）落入阿里云向量数据库（DashVector）。
3. **召回与交叉重排（Retrieval & Rerank）**
   循环遍历标书 A 的每一个切片文本块。在 DashVector 中搜索出标书 B 里极度相似的 Top-15 文本块。随后调用百炼 `gte-rerank` 接口，要求其对这 15 个候选项进行交叉对比与精细打分，剔除得分 `< 0.85` 的噪音干扰片段。
4. **大模型法官裁决（LLM Agent Judge）**
   把重排后截获的“高危嫌疑文本对”送入 Qwen-Max/Plus。系统在 Prompt 中设定专业严谨的“审计/法务专家”角色。大模型负责剔除合规的引用重合（如两者在此段都引用了公卫安全标准），并对实质性抄袭（如核心参数雷同仅替换主体名）给出违规等级与详细宣判理由。

## 3. 技术栈与现有工程适配

作为后续实际开发编码落地的依据，技术选型尽量复用项目中已引入的技术库，降低系统复杂度。

| 模块层级 | 技术选型 | 在本工程中的落地方案与职责说明 |
|---|---|---|
| **应用与路由** | Spring Boot 3.4 | 统一对外提供对比与查询的 REST API，承载后端核心调度逻辑。 |
| **底座与模型 SDK** | Spring AI Alibaba <br/> DashScope SDK | 当前项目 `pom.xml` 已包含相关依赖。将直接使用其中对 VectorStore 与大模型 API 的流式对话包装能力及 `Message` Builder 构造机制。 |
| **非结构化文档解析** | Docling / 阿里云文档智能提取 | 推荐引入轻量级或在线的版面分析接口来还原 PDF 中的表格。建议封装独立的 `PdfParserService`。 |
| **文档语义切块** | Spring-AI DocumentSplitter | 定制开发 `TokenTextSplitter` 策略，保障每个切块携带章节 Title 等 Metadata 元数据以供溯源（标书 A 的第 X 章被标书 B 的第 Y 章抄袭）。 |
| **向量存储检索** | 阿里云 DashVector | 需要引入 `dashvector-java-sdk`，将 `bid_demo_collection` 索引集合映射至后端代码完成数据的 Insert 和 nearest neighbor 检索。 |
| **基座语言模型** | 通义千问 Qwen-Max / Plus | 复用/扩展现有的 `AgentChatService` 或 `QwenService`，定制针对查重的专属 System Prompt 提供给大模型作为判决准则。 |

## 4. 后端核心模块详细设计

### 4.1 标书审查大模型 Agent 服务 (`BidReviewAgentService`)
基于现有的 `AgentChatService` 增加新的交互场景：
- **System Prompt 扩充：** 在 `SystemPromptManager` 增加 `BID_PLAGIARISM_REVIEW_PROMPT`。
  *逻辑重点：* “你是一名专业的政企招投标审计专家。请对比提供的【待查文本A】与【基准文本B】。注意识别合法引用的法规条文并排除它们。如果属于核心业务参数、价格造假或明显的模板套用洗稿行为，请如实指出并输出违规级别和缘由。必须严格按结构化 JSON 输出。”

- **结构化内容输出：**
  利用大模型的 `JSON Mode` (或者在 `GenerationParam` 中指定 ResultFormat)，强制 Qwen 模型输出标准数据结构，避免返回冗余废话影响双端展示：
  ```json
  {
    "is_plagiarism": true,
    "risk_level": "HIGH", 
    "similarity_type": "技术参数完全重合 / 业务逻辑洗稿 / 模板套用",
    "reason": "片段A与片段B在服务器集群技术规格描述上达到98%以上雷同，仅替换了供应商缩写，且特殊金额数字出现异常吻合，不属于公开标准法规引用范畴，极高疑似围标串标行为。"
  }
  ```

### 4.2 API 接口规划定义 (BidReviewController)
查重对比往往是消耗几分钟甚至几十分钟的异步过程，接口必须采用异步任务设计：

1. **POST `/api/bid/compare`**
   - **入参：** `MultipartFile fileA` (标书A待查), `String referenceBidId` (基准标书库ID或文件)
   - **响应：** 快速返回系统级任务 ID (`taskId`)，不阻塞前端页面。
2. **GET `/api/bid/task/{taskId}/progress`**
   - **功能：** 供前端轮询查询（或 SSE 推送）当前对比的进度百分比和可视化状态文案（如：“正在对比标书 A 的第 5 章节文本...”，“正在调用 Qwen-Max 审查嫌疑雷同项...”）。
3. **GET `/api/bid/report/{taskId}`**
   - **功能：** 等待分析完成后，下发完整的汇总分析报告与双向高亮比对数据点。

## 5. 前端交互体验设计思路 (Vue / Vite)

为了赋予该医疗合规底座“科技感”和业务直观性，前端页面需具备以下表现层：

1. **异步进度与分析感知界面：**
   查重过程需要通过类似骨架屏和流水线步骤条的形式，实时反馈后端的解析动态。不要让页面处于假死与无休止的 Loading 状态。
2. **双栏沉浸式高亮对比视图（核心）：**
   在结果详情页下半部使用左右对照的双栏阅读器：左侧为标书 A 相关段落，右侧为标书 B。
   使用不同颜色的 Badge（标签）或高亮背景色标记雷同发生的确切位置，大模型的分析原因/判决悬浮在这些色块旁，即便不懂后台实现的老板/法务人员也能一眼看懂：“哪里疑似抄袭，为什么说是抄袭”。
3. **宏观雷同率雷达与汇总呈现：**
   通过图表（如 ECharts），展现整本标书各章节的综合雷同率波动分布。

## 6. 接下来的落地实施步骤 (建议路径)

为了稳步推进场景一落地，建议按照以下里程碑执行：
- **Step 1 (基础设施构建)：** 跑通 DashVector (阿里云端向量库) 在现有 Spring Boot 项目内的读写测试和千问 Embedding / Rerank 接口连通。
- **Step 2 (文本加工流水线)：** 开发处理 `PDF/Docx -> Markdown` 和 `Semantic Chunking` (基于大语言模型框架或 Spring AI 提供的 Splitter) 操作。
- **Step 3 (审计判定与报告拼装)：** 端到端编写 `BidReviewService` 获取 RAG 检索结果，喂入千问获得 JSON，组装整个报告结构模型。
- **Step 4 (前端集成)：** 基于现有的 Vue 工程搭建专用的 `/bid-review` 路由视图并联调渲染上述复杂的比对结果。
