# 医疗agent相关场景

## 场景一：超长标书文件的高度雷同与语义查重

### 1. 整体系统工作流 

系统整体分为四个核心阶段：**文档工程阶段 ➔ 索引构建阶段 ➔ 混合检索与重排阶段 ➔ 大模型裁决阶段**。

- #### 阶段一：基础设施与云端联调

  **目标：跑通百炼 API，证明云端算力链路可用，消除技术盲区。** **时间预算：第 1-2 天**

  在这个阶段，我们先不碰复杂的标书文件，而是用最简单的字符串测试所有的底层接口。

  - **Task 1.1：百炼 API 与 LLM 基础对话调通**
    - **动作：** 获取阿里云百炼的 API Key，配置本地环境变量。使用兼容 OpenAI 格式的 Python SDK，向 `qwen-max` 发送一句简单的测试 prompt（如：“你好，请扮演一名严谨的审计专家”）。
    - **验收标准：** 终端成功打印出 Qwen-Max 的流式回复，无网络报错。
  - **Task 1.2：向量化（Embedding）与重排（Rerank）接口双测**
    - **动作：** 准备两句简短的话（例如：A="本项目总预算为十万元"，B="该工程整体资金额度达100,000元"）。调用 `text-embedding-v3` 将其转化为两个高维向量。接着，将这两句话输入 `gte-rerank` 接口获取相似度打分。
    - **验收标准：** 成功获取 1024 维度的浮点数数组；重排接口返回 >0.9 的高分。
  - **Task 1.3：云端向量库（DashVector）连通性测试**
    - **动作：** 在阿里云控制台免费开通 DashVector 实例。通过 Python 官方 SDK 建立一个名为 `bid_demo_collection` 的测试集合。尝试插入刚才生成的向量，并通过简单的关键词或向量进行一次 Retrieve（查询）。
    - **验收标准：** 成功 Insert 数据并能 Retrieve 出对应的文本块。

  ------

  #### 🟡 阶段二：文档流水线与底库构建

  **目标：将沉重的非结构化 PDF 标书，转化为大模型和向量库能够理解的优质结构化数据。** **时间预算：第 3-5 天**

  这是整个系统最脏最累的活，数据质量直接决定查重的准确率。

  - **Task 2.1：多模态文档解析（PDF -> Markdown）**
    - **动作：** 使用百炼的文档智能抽取 API（或轻量级开源替代方案如 Docling），将一份包含表格和红头文件的测试标书 B（建议精简到 20 页以内）进行解析。
    - **验收标准：** 得到一份层级分明（包含 `#`, `##` 标题）且表格没有乱码的 Markdown 文件。
  - **Task 2.2：智能语义切块（Semantic Chunking）**
    - **动作：** 引入大模型开发框架 LlamaIndex。编写脚本，使用 Markdown 节点解析器（MarkdownNodeParser），按照文档的章节逻辑将其切分为多个 Chunk（每块约 500-800 tokens，保留少许重叠区）。
    - **验收标准：** 打印切分后的数组，确保同一条款或表格没有被生硬地从中间劈断。
  - **Task 2.3：自动化建库脚本（Ingestion Pipeline）**
    - **动作：** 编写 `build_index.py`。该脚本负责：读取 Markdown -> 语义切块 -> 批量调用百炼 Embedding -> 携带着元数据（如章节名、页码）存入 DashVector。
    - **验收标准：** 运行脚本后，标书 B 被完美切片并安全沉淀在云端数据库中，随时准备被检索。

  ------

  #### 🟠 阶段三：混合检索与大模型裁决法庭

  **目标：用标书 A 去“碰撞”标书 B，并让大模型给出严谨的法律审查级结论。** **时间预算：第 6-9 天**

  这是这个系统最具技术含量的地方，我们要让机器拥有类似审查民法典合同条款般的严谨度。

  - **Task 3.1：多路召回与重排过滤引擎**
    - **动作：** 编写 `retriever.py`。解析标书 A 并循环它的每一个 Chunk。去 DashVector 中召回最相似的 Top-15 标书 B 的 Chunk。随后，立刻调用百炼的 `gte-rerank` 接口对这 15 个组合进行重新交叉打分。
    - **验收标准：** 成功过滤掉大量低相关性的“噪声”片段，仅保留得分 `> 0.85` 的高危“嫌疑组合”。
  - **Task 3.2：Prompt 工程与大模型“法官”宣判**
    - **动作：** 编写极度严密的 System Prompt，定义判定规则（什么是废标级的抄袭、什么是国家标准的合法引用）。利用 Qwen-Max 的强推理能力，将前一步筛选出的高危文本对塞入 Prompt 中。
    - **验收标准：** 大模型能够稳定地剔除“假阳性”（例如两家公司都引用了相同的招投标法规），并输出标准的 JSON 格式：`{"is_plagiarism": true, "similarity_type": "核心业务逻辑重合", "reason": "..."}`。
  - **Task 3.3：全局报告聚合**
    - **动作：** 将循环比对后产生的所有 JSON 结果进行合并，计算出整体的“疑似雷同率”，并生成一份总体的分析报告数据结构。

  ------

  #### 🔴 阶段四：前端交互体验与 Demo 

  **目标：包装底层技术，呈现一个优雅、直观、高商业价值的企业级产品界面。** **时间预算：第 10-14 天**

  - **Task 4.1：Streamlit 极速前端搭建**
    - **动作：** 使用 Python 的 Streamlit 库搭建 Web UI。实现两个文件上传组件（标书 A 和 标书 B），一个醒目的“启动深度审计”按钮。
    - **验收标准：** 页面能够在浏览器中正常加载，文件上传功能畅通。
  - **Task 4.2：过程可视化与进度反馈（关键加分项）**
    - **动作：** 在执行耗时的比对循环时，千万不要让页面假死。利用 Streamlit 的进度条和状态组件（`st.status`），实时打印当前的执行状态（如：“正在重排第 3 章片段...”、“正在呼叫 Qwen-Max 进行法务裁决...”）。
    - **验收标准：** 查重过程充满“科技感”，用户能清楚感知到系统背后的复杂工作流。
  - **Task 4.3：双栏高亮对比视图**
    - **动作：** 查重结束后，在页面下半部分渲染一张清晰的对比表格或分栏卡片。左侧展示标书 A 的雷同段落，右侧展示标书 B，并在下方用不同颜色的标签（Badge）标出大模型给出的判定理由和危险等级。
    - **验收标准：** 任何人（即使是不懂技术的业务老板）看一眼就能明白哪里抄了、为什么判抄袭。

## 场景二：合同文件的 AI 预审核

**难点剖析：** 合同审核对“准确率”要求极高，大模型喜欢“脑补”（幻觉）是致命伤。此外，审核往往需要对照企业内部的标准模板或法律法规。

**技术落地方案（信息抽取 + 规则引擎 + 角色扮演 Prompt）：**

1. **关键信息抽取（NER / Information Extraction）：** 首先让 LLM 提取合同的核心要素（如：甲乙方主体、合同金额、付款节点、违约金比例、管辖法院等），将其结构化为 JSON。
2. **构建审核规则库（Knowledge Base / Rules）：** 合同预审通常是带有明确倾向性的（比如：我是甲方，绝对不能接受在乙方所在地法院起诉）。这些规则需要被写成明确的 Checklists。
3. **分步审核（Step-by-step Verification）：** 不要让模型一次性看完合同就给出评价。针对每一个 Checklist 条目（如：“检查违约条款是否对等”），单独检索合同中相关的条款，让 LLM 扮演**资深法务**的角色进行单点判定。
4. **风险高亮：** 将 LLM 找出的风险点在原文中进行高亮映射，并给出修改建议。

## 场景三：医疗行动耗材与药品合规风险预警

**难点剖析：** 这是一个典型的**“结构化数据分析 + 领域知识 + 规则/预测模型”**的场景。大模型并不擅长直接做精准的数值预测（比如准确算出需要几个手术钳）。

**技术落地方案（预测/规则模型 + 知识图谱 + LLM 归因）：** 这与开发药品监管智能体的逻辑有异曲同工之妙。底层需要的是严谨的数据和规则，表层需要的是 LLM 的推理和解释能力。

1. **基线预测（传统 ML 或 规则引擎）：** * 针对特定手术（如阑尾切除术），你需要一套历史结构化数据。
   - 可以使用传统的机器学习模型（如 Random Forest、XGBoost）或基于医疗知识图谱的确定性规则，来计算出某种耗材的“标准消耗区间”（比如预测使用 3-5 个）。
2. **异常检测（Anomaly Detection）：** 当实际输入（10个）偏离基线（3-5个）时，系统触发警报机制。
3. **大语言模型分析与归因（Agent 介入）：** * 一旦触发预警，Agent 会自动调取该场手术的电子病历（EMR）、医生级别、患者并发症等信息。
   - LLM 结合这些上下文进行推理：“虽然标准预估是5个手术钳，但该患者有严重的腹腔粘连病史，且手术时长超标，10个的消耗量虽然异常但存在合理医学解释，建议提请人工复核。”
