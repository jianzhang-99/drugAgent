# 医疗监管AI智能体 - 场景一：超长标书文件雷同与语义查重

自动化识别医疗招投标过程中的围标、串标及标书抄袭行为，解决传统查重工具难以处理数百页复杂排版长文档、“洗稿”替换以及法规标准合法引用误判等痛点。

## 核心能力 (Features)

- **超长文档解析与切片：** 突破大模型上下文限制，将数百页PDF/Word智能切分入库。
- **高精度长文本向量检索：** 基于阿里云 DashVector 与 text-embedding-v3 进行高维语义匹配。
- **防止误判的交叉重排：** 引入 gte-rerank 接口对初步检索结果进行二次精细打分降噪。
- **AI 法务级裁决引擎：** 扮演审计专家角色，精准排除“合法法规引用”，直击核心业务参数造假。

## 1. 总体系统架构设计 (Architecture)

本场景采用成熟的 **高级 RAG (Retrieval-Augmented Generation) 架构 + 混合检索与重排 + 大模型裁决** 机制。

### 工作流步骤

1. **文档抽取 (Ingestion)：** 将排版复杂的标书（PDF/DOCX）抽取为保留标题、表格结构的 Markdown。
2. **切分建库 (Chunking & Embedding)：** 利用语义切分器（Semantic Chunking）将文本分割为 500-800 Token 的段落，转化为高维向量并存入 DashVector 向量库，同时保留章节、页码等元数据 (Metadata)。
3. **混合检索 (RetrieVAL & Rerank)：** 循环待查标书的每一段落，在库中召回最相似的 Top-15 文本块。调用 `gte-rerank` 接口对这 15 个候选项进行重排，滤除相似度得分 `< 0.85` 的噪音。
4. **AI 裁决 (Agent Judge)：** 将高危嫌疑文本对送入 Qwen 大模型。大模型根据内置的“审计专家”系统提示词，判断是否为合法的标准引用，或输出包含判定理由和危险等级的严谨裁决。

## 2. 后端核心模块详细设计 (Backend Design)

当前项目基于 `Spring Boot 3.4.4` 和 `Spring AI Alibaba`。

### 核心对象：标书审查 Agent (`BidReviewAgentService`)

此服务负责统筹大模型的直接交互。

**系统提示词 (System Prompt) 示例：**
> 你是一名专业的政企招投标审计专家。请对比提供的【待查文本A】与【基准文本B】。注意识别合法引用的法规条文并排除它们。如果属于核心业务参数、价格造假或明显的模板套用洗稿行为，请如实指出并输出违规级别和缘由。必须严格按结构化 JSON 输出。

**预期输出格式 (JSON Mode)：**
```json
{
  "is_plagiarism": true,
  "risk_level": "HIGH", 
  "similarity_type": "技术参数完全重合 / 业务逻辑洗稿 / 模板套用",
  "reason": "片段A与片段B在服务器集群技术规格描述上达到98%以上雷同，仅替换了供应商缩写，且特殊金额数字出现异常吻合，属于极高疑似围标行为。"
}
```

### API 接口规划定义 (`BidReviewController`)

由于查重是耗时（几分钟至半小时）的 I/O 密集型任务，API 设计必须是**异步**的。

| 接口端点 (Endpoint) | 方法 (Method) | 说明 (Description) | 返回值 (Returns) |
|---------------------|---------------|--------------------|------------------|
| `/api/bid/compare` | `POST` | 提交标书对比任务。接收 `fileA` 和基准 `referenceBidId`。 | 返回系统级任务 ID (`taskId`) |
| `/api/bid/task/{taskId}/progress` | `GET` | 轮询查询任务进度。 | 返回进度百分比及当前动作文案 (`statusText`) |
| `/api/bid/report/{taskId}` | `GET` | 任务完成后获取最终详细报告。 | 包含整体雷同率及具体雷同段落坐标和分析理由的 JSON |

## 3. 前端交互体验设计 (Frontend UI)

为了提供直观、专业的企业级审查体验，前端 (Vue 3 / Vite) 需实现以下视图：

- **进度感知大屏：** 抛弃静态 Loading 圈。使用步骤条和动态文本（如：“正在对比第 5 章...”、“呼叫 Qwen-Max 进行法务审核...”）实时反馈后台流处理进度。
- **双栏沉浸式高亮视图：** 详情页下半部展示对照阅读器。
  - 左侧展示标书 A 文本，右侧展示标书 B 对应文本。
  - 核心雷同区域使用高亮色块标记。
  - 大模型的分析结论以气泡卡片 (Popover) 或徽标 (Badge) 形式悬浮于高亮区域旁。
- **宏观分析雷达：** 侧边或顶部提供各章节综合雷同率分部的雷达图/柱状图。

## 4. 下一步开发实施路径 (Next Steps)

1. **第 1 阶段：** 配置连通阿里云 DashVector 向量数据库，跑通千问 `text-embedding-v3` 和 `gte-rerank` 接口。
2. **第 2 阶段：** 搭建非结构化文档转 Markdown 并进行语义切分 (Semantic Chunking) 的预处理脚本或 Java 服务。
3. **第 3 阶段：** 使用 `Spring AI Alibaba` 框架，在 `BidReviewAgentService` 开发 RAG 取回、Prompt 组装和向 Qwen-Max 发起 JSON 裁决请求的核心业务逻辑。
4. **第 4 阶段：** 编写异步 Controller 层，在 Vue 前端开发双栏对比高亮组件及相应的交互视图。
