# 02 学习路径与周计划

## 1. 四周学习与开发总览

| 周次 | 学习重点 | 产出物 | 通过标准 |
|---|---|---|---|
| 第1周 | 跑通基础设施与异步任务骨架 | 可创建任务并查进度 | 能返回 `taskId` + 任务状态 |
| 第2周 | 文档切片、向量检索、重排 | 候选雷同片段列表 | 能得到 TopK 可疑片段 |
| 第3周 | 大模型裁决与报告组装 | 结构化报告 JSON | 能区分合法引用与疑似抄袭 |
| 第4周 | 前端页面与联调验收 | 上传 + 进度 + 报告展示 | 可完整演示场景一 |

## 2. 每周任务拆解（按 2-8 小时粒度）

### 第1周：基础与任务引擎

| 任务 | 预计工时 | 依赖 | 完成标准 |
|---|---:|---|---|
| 阅读现有代码与配置 | 2h | 无 | 画出当前调用链路 |
| 新建 `bidreview` 包结构 | 2h | 无 | 包结构可编译 |
| 定义任务实体与状态机 | 4h | 包结构 | 有 `taskId`/状态/进度对象 |
| 实现 `POST /api/bid/compare` | 4h | 状态机 | 提交成功返回 `taskId` |
| 实现 `GET /api/bid/task/{taskId}/progress` | 4h | 状态机 | 能轮询进度 |

### 第2周：检索链路

| 任务 | 预计工时 | 依赖 | 完成标准 |
|---|---:|---|---|
| 实现文本解析（先 `.md/.txt`） | 6h | 第1周完成 | 能稳定提取文本 |
| 实现语义切片服务 | 8h | 文本解析 | 可输出 500-800 token 切片 |
| 实现向量入库与查询 | 8h | 切片服务 | 可检索 TopK 候选 |
| 接入重排模型与阈值过滤 | 8h | 向量查询 | 可输出高危候选对 |

### 第3周：AI 裁决与报告

| 任务 | 预计工时 | 依赖 | 完成标准 |
|---|---:|---|---|
| 新增场景一 System Prompt | 3h | 无 | Prompt 可复用 |
| 实现裁决服务（JSON 输出） | 8h | 候选片段 | 返回结构化字段 |
| 设计报告模型与聚合逻辑 | 8h | 裁决服务 | 生成完整报告 |
| 实现 `GET /api/bid/report/{taskId}` | 4h | 报告模型 | 可返回最终报告 |

### 第4周：前端与验收

| 任务 | 预计工时 | 依赖 | 完成标准 |
|---|---:|---|---|
| 新增 `/bid-review` 路由与页面 | 4h | 后端接口可用 | 页面可访问 |
| 上传与进度组件开发 | 8h | 任务接口 | 可实时看到进度 |
| 双栏结果与高亮组件 | 12h | 报告接口 | 能展示雷同片段 |
| 联调与回归测试 | 8h | 全链路 | 通过验收清单 |

## 3. 学习顺序建议（避免卡住）

1. 先完成“异步任务框架”，再做“AI 能力”
2. 先支持文本格式，再扩展 PDF/DOCX
3. 先用 Mock 报告打通前端，再联真接口
4. 每天结束必须留下一条可验证结果（接口、日志或截图）

## 4. 关键依赖路径

`任务骨架 -> 文本切片 -> 向量检索 -> 重排 -> LLM裁决 -> 报告 -> 前端展示`

其中前 5 步是关键路径，任一步不稳定都会影响最终结果。
