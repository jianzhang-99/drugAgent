# 场景一标书审查 Workflow 初步设计文档

## 1. 文档目的

本文档用于说明“标书审查”场景的初步技术设计方案，明确系统边界、模块拆分、核心流程、数据结构及 MVP 实施范围，为后续详细设计、研发实施和测试验收提供依据。

## 2. 适用范围

本文档适用于“场景一：标书审查”能力建设，主要面向以下参与角色：

- 产品经理：用于确认能力范围与阶段目标
- 架构师/研发工程师：用于拆分服务、定义接口与实施方案
- 测试工程师：用于理解输入输出和验收重点
- 项目管理人员：用于识别 MVP 范围及后续演进方向

## 3. 设计目标

### 3.1 业务目标

系统需支持对投标文件进行自动化审查。

`V1` 阶段以围标、陪标相关风险线索识别为主，重点识别以下风险信号：

- 联系人、电话、邮箱、地址等主体信息异常一致
- 团队成员、岗位、资质等信息重复或交叉复用
- 报价规律异常、报价梯度异常
- 模板化痕迹、罕见错误共现等高风险特征

`V2` 阶段在此基础上继续增加全文文本语义分数，用于识别文本雷同、洗稿式改写和结构高度同源等风险。

系统应该输出一份风险报告供前端展示，其基础特征工程和风险原因应与本文档中定义的结构化对象保持一致。

### 3.2 技术目标

- 建立可回溯、可解释的审查流水线
- 采用固定流程编排，优先落地规则驱动的围标风险识别能力
- 为后续增加全文语义比对、批量审查、历史库比对、关系图谱分析预留扩展能力

### 3.3 设计原则

- 可控性优先：核心判定由规则、算法和结构化流程完成
- 可解释性优先：所有高风险结果必须关联可定位证据
- 可演进性优先：V1 先做规则驱动闭环，V2 再引入全文语义评分能力
- 低误报优先：通过豁免规则、章节权重和证据融合降低误判

## 4. 系统边界

### 4.1 系统负责内容

- 接收标书审查任务
- 解析上传文档并提取结构化信息
- 执行围标规则检测、误报豁免和风险报告生成
- 输出风险等级、高风险片段、证据链和审查报告
- 支持人工复核工作台查看证据和复核结果

### 4.2 系统不负责内容

- 不直接认定违法违规
- 不替代法务、审计或监管人员出具最终结论
- 不在 MVP 阶段覆盖 OCR、复杂版面恢复、多模态解析、跨项目关系图谱全量分析

## 5. 总体架构

系统采用“上层通用 Agent + 场景编排 Agent + 专业分析服务 + 报告生成服务”的分层架构。`V1` 阶段核心思想是以结构化解析和规则分析为主，先构建稳定可解释的审查闭环；后续再逐步引入语义比对能力。

```text
上层通用 Agent
-> 场景路由
-> TenderReview 场景 Agent（编排器）
-> 文档解析引擎
-> 围标特征规则引擎
-> 误报豁免引擎
-> 证据编排与报告 Agent
-> 人工复核工作台
```

## 6. 架构设计结论

本场景建议采用以下结构：

- 1 个场景编排与报告 Agent
- 3 个专业 Workflow/分析服务

对应职责如下：

- 编排与报告 Agent：负责流程控制、任务状态管理、服务调用、结果聚合，以及将结构化结果转写为审查说明、复核建议和对外报告
- 专业分析服务：负责文档解析、规则命中、豁免判断等确定性分析

## 7. 核心处理流程

标书审查场景采用固定 DAG 流程，不使用大模型执行核心判断。`V1` 阶段以围标规则驱动为主，全文语义评分能力在 `V2` 再接入。

```text
任务接收
-> 文档解析
-> 结构化提取
-> 围标规则命中
-> 误报豁免
-> 风险融合
-> 证据编排
-> 报告生成
```

流程说明如下。

### 7.1 任务接收

任务接收阶段的目标是完成一次审查任务的初始化，确保后续流程具备统一的任务上下文。

输入内容包括：

- 用户提交的审查请求
- 待审查文档列表
- 业务参数，例如审查类型、比对范围、优先级
- 用户身份、角色和权限信息

处理逻辑建议如下：

- 生成唯一 `caseId`，创建审查任务主记录
- 校验上传文件格式、大小、数量和完整性
- 记录任务创建人、所属组织、提交时间和任务参数
- 根据用户权限确定可访问文档范围和结果可见范围
- 将任务状态初始化为“待解析”或“处理中”
- 将原始文件统一存储到对象存储或文件服务，并登记存储路径

输出结果包括：

- 审查任务主记录
- 文档元数据记录
- 可供后续流程使用的任务上下文对象

实现建议：

- 在数据库中拆分 `case_task`、`case_document`、`case_operation_log` 三类表
- 对文件上传和任务创建做幂等处理，避免重复提交
- 预留异步处理能力，避免大文件任务阻塞接口响应

### 7.2 文档解析

文档解析阶段的目标是将非结构化标书内容转换为可计算、可定位的内容块。

输入内容包括：

- 原始标书文件
- 文件基础信息，例如文件名、类型、页数、大小

处理逻辑建议如下：

- 按文件类型调用对应解析器，MVP 阶段优先支持 `docx`
- 提取文档基础属性，例如标题、页数、段落数、表格数
- 解析章节标题树，建立文档层级结构
- 拆分正文段落、表格、附件说明等内容块
- 提取段落顺序、表格位置、章节归属等上下文信息
- 为每个内容块生成锚点信息，例如页码、章节号、段落号、表格号

输出结果包括：

- 文档基础信息
- 章节树
- 段落块列表
- 表格块列表
- 初始锚点信息

实现建议：

- 解析层尽量保持“忠实还原”，不要在该阶段做过多业务判断
- 对解析失败、空文档、异常表格等情况输出标准错误码
- 为后续证据对照页面保留原始顺序和位置信息

### 7.3 结构化提取

结构化提取阶段的目标是将解析结果标准化，生成统一的数据对象，供后续规则分析、风险融合和报告模块复用。

输入内容包括：

- 文档解析结果
- 字段抽取规则
- 章节分类规则

处理逻辑建议如下：

- 对段落内容做清洗，例如去除多余空白、统一标点、规范编码
- 识别并标注块类型，例如 `paragraph`、`table`、`field`
- 提取关键业务字段，例如联系人、电话、邮箱、报价、团队成员、服务承诺、风险控制条目
- 为每个块补充章节路径、文档编号、块编号、来源锚点
- 对字段值做标准化处理，例如手机号归一化、邮箱小写化、金额格式化
- 生成供规则分析使用的特征字段，例如关键词、tokens、字段标签和标准化键

输出结果包括：

- 统一 `Block` 集合
- 统一 `Field` 集合
- 标准化字段字典和特征数据

实现建议：

- 结构化提取应形成稳定 schema，避免后续模块直接依赖原始解析格式
- 字段抽取可结合规则、模板和 LLM 辅助校正，但最终结果需落为结构化对象
- 对提取失败字段保留空值和置信度，便于后续回溯

### 7.4 围标规则命中

围标规则命中阶段的目标是识别文本相似之外的协同行为风险特征。

输入内容包括：

- 结构化字段数据
- 多文档之间的比对关系
- 围标规则配置

处理逻辑建议如下：

- 对联系人、电话、邮箱、地址执行一致性和相似性检测
- 对团队成员、岗位、证书、资质执行重复度和交叉复用分析
- 对报价明细执行梯度异常、区间异常、同构报价等规则检测
- 对罕见错字、异常格式、特定模板痕迹执行共现分析
- 对每条命中规则记录命中值、触发条件、关联文档和证据锚点

输出结果包括：

- 围标规则命中清单
- 每条命中的风险权重
- 关联证据索引

实现建议：

- 围标规则建议配置化管理，支持后续持续补充
- 每条规则应具备编号、说明、优先级和可解释输出
- MVP 阶段优先做高确定性规则，避免一开始铺得过宽

### 7.5 误报豁免

误报豁免阶段的目标是剔除“高风险特征明显但实际合理”的内容，提升整体结果可信度。

输入内容包括：

- 围标规则结果
- 白名单库、法规库、模板库

处理逻辑建议如下：

- 识别法规条文、标准引用、招标模板固定语等内容
- 判断命中内容是否出现在低风险章节，例如公司介绍、常规资质说明
- 对通用表达、标准表述、白名单模板内容进行降权
- 将豁免结果与规则命中项做交叉校验，避免误将模板内容抬升为高风险
- 对豁免项记录原因、命中库和降权幅度

输出结果包括：

- 豁免命中清单
- 调整后的分项分数
- 误报过滤说明

实现建议：

- 误报豁免不是简单删除，而是建议采用“降权 + 记录原因”的方式处理
- 法规库、模板库应支持独立维护和版本化
- 该模块需要结合真实样本持续调优，是系统效果的关键抓手

### 7.6 风险融合

风险融合阶段的目标是把规则命中结果和豁免结果统一汇总为可理解的风险结论。

输入内容包括：

- 围标规则命中结果
- 豁免结果
- 风险权重与阈值配置

处理逻辑建议如下：

- 先处理高优先级硬规则，例如相同电话、相同邮箱、异常报价梯度
- 再结合章节权重、规则权重和豁免结果计算综合分值
- `V1` 阶段优先输出 `collusion` 和 `overall`
- 根据分值区间映射风险等级，例如低、中、高
- 生成结构化风险原因码和解释项

输出结果包括：

- 各风险类型分值
- 各风险类型等级
- 风险原因项和关联证据

实现建议：

- 风险融合逻辑不要写死在代码分支中，建议参数化配置
- 高风险判定应同时考虑“命中强度”和“证据充分度”
- `V2` 阶段可逐步接入全文语义分数和基于样本的权重学习，但 `V1` 阶段优先采用规则化融合

### 7.7 证据编排

证据编排阶段的目标是把机器计算结果转换成可供人工复核的证据链。

输入内容包括：

- 风险融合结果
- 围标规则命中和豁免记录
- 原始锚点信息
- 结构化 `Block` 和 `Field`
- 风险原因码与规则编号映射关系

处理逻辑建议如下：

- 以风险类型和风险原因码为主线组织证据，不直接堆叠原始命中结果
- 为每条高风险结论建立 `reasonCode -> evidenceIds -> sourceHitIds` 的映射关系
- 从 `RuleHit`、`ExemptionHit`、`Block`、`Field` 中抽取可直接复核的证据片段
- 组织原文片段、命中字段和值、命中规则说明、豁免说明和来源对象 ID
- 补充锚点信息，例如文档名称、章节路径、页码、段落号、表格号、字段位置
- 将多个证据片段按风险主题分组，例如“联系人重合”“团队重复”“报价异常”
- 在分组内区分主证据和辅助证据，优先保留最能支撑结论的证据项
- 对重复证据进行合并去重，避免同一字段、同一锚点被多个规则重复展示
- 按风险强度、证据充分度和人工可读性对证据排序
- 输出标准化证据对象，再由展示层转换为前端工作台所需视图对象

输出结果包括：

- 证据项清单
- 证据分组结果
- 风险原因码到证据链的映射结果
- 前端可展示的证据对照数据

实现建议：

- 证据编排要面向人工复核体验设计，重点是“看得懂”和“点得回去”
- 每条证据应尽量独立成立，避免只给分数不给依据
- 证据对象建议支持原文高亮、字段高亮和跨文档跳转
- 证据编排层建议输出统一 `EvidenceItem` 结构，不直接耦合某个页面组件
- 每个风险原因码至少应绑定一条主证据，必要时补充多条辅助证据
- 证据结果应保留规则版本、豁免版本和生成时间，便于后续复核追溯

### 7.8 报告生成

报告生成阶段的目标是形成可阅读、可导出、可追问的审查输出报告。

输入内容包括：

- 风险融合结果
- 证据编排结果
- 任务基础信息
- 文档基础统计信息
- 审查策略版本和规则版本信息

处理逻辑建议如下：

- 基于结构化 `RiskScore`、`EvidenceItem` 和任务元数据组装报告，不直接依赖原始解析结果拼装页面
- 生成总体审查摘要，包括审查范围、文档数量、主要结论、风险分布和建议优先复核项
- 输出风险清单，包括风险类型、风险等级、风险原因码、核心证据和处置建议
- 输出高风险片段列表、跨文档对照说明和重点复核路径
- 针对页面展示、导出文档、接口返回和追问解释生成不同视图，但底层共享统一报告对象
- 在需要自然语言说明的部分使用 LLM 生成审查表述、复核建议和用户追问回复
- 对 LLM 生成内容附加引用证据和来源对象，避免脱离证据自由生成
- 为报告结论补充版本信息、生成时间和任务上下文，便于后续复核追溯

输出结果包括：

- 页面展示版审查结果
- 导出版审查摘要
- 结构化报告对象
- 面向追问的解释结果对象

实现建议：

- 报告生成应基于结构化结果，不应让 LLM 脱离证据自由发挥
- 导出内容建议区分“管理摘要”和“证据明细”两层
- 页面版、导出版和问答版建议基于统一 `ReviewReport` 对象派生，避免多套字段重复维护
- 报告中的每条高风险结论应能回链到 `reasonCode`、`evidenceIds` 和原始命中记录
- 报告摘要应突出“先看结论、再看证据、最后看建议”的阅读顺序，降低人工复核成本
- 后续可支持用户追问，例如“为什么这段是高风险”“该结论基于哪些证据”

### 7.9 V2 全文语义评分扩展

在 `V1` 规则驱动闭环稳定后，`V2` 阶段增加全文语义评分能力，用于补充“规则未命中但文本高度同源”的风险识别。

输入内容包括：

- 标准化后的 `Block`
- 章节信息和锚点信息
- 语义检索模型和精排模型配置

处理逻辑建议如下：

- 对段落和表格内容建立向量索引
- 在同类章节或限定范围内进行候选召回
- 对召回候选执行语义相似度计算和句对级精排
- 将语义分数作为新增风险特征接入风险融合层
- 保留语义命中的证据片段和对应锚点

输出结果包括：

- 候选内容对列表
- 语义分数结果
- 语义证据项

实现建议：

- `V2` 新增能力应作为独立模块接入，不影响 `V1` 规则链路稳定性
- 语义分数只作为增强特征，不直接替代规则命中
- 优先在高价值章节中试点，例如技术方案、服务承诺、风险控制章节

## 8. 模块设计

### 8.1 上层通用 Agent

上层通用 Agent 负责统一入口层能力，不参与具体审查计算，主要职责包括：

- 会话管理
- 用户意图识别
- 场景路由
- 权限校验
- 任务创建
- 结果问答与追问解释

### 8.2 TenderReview 场景 Agent

TenderReview 场景 Agent 是本场景的核心编排器，仅负责流程控制，不直接基于全文做最终风险判断。

主要职责包括：

- 接收上层任务并初始化 Case
- 按固定流程调度解析、比对、规则和报告服务
- 维护任务状态、阶段结果和异常重试
- 对外输出结构化审查结果

建议职责边界：

- 不直接读取全文后由 LLM 一次性给出风险结论
- 不承担规则计算、相似度计算等分析逻辑
- 不绕过专业服务直接生成审查结论

### 8.3 文档解析引擎

文档解析引擎决定后续分析效果，是系统的基础能力模块。

解析结果至少应覆盖以下结构层：

- 文档基础信息
- 章节标题树
- 段落列表
- 表格列表
- 关键字段
- 锚点信息

建议抽取的重点字段包括：

- 联系方式
- 团队角色与成员信息
- 报价字段
- 服务承诺
- 风险控制相关字段

设计要求：

- 解析结果必须保留“可回溯锚点”
- 锚点至少应支持页码、章节号、段落号、表格号中的一种或多种组合
- 文本块、表格块、字段块需统一编码，便于后续关联证据

### 8.4 围标规则引擎

`V1` 阶段的核心分析模块为围标规则引擎，不默认引入全文语义评分。

该模块用于识别非纯文本抄袭但存在高度同源或协同行为的风险，建议特征包括：

- 联系人、电话、邮箱相近或重复
- 团队角色和人员构成重复
- 报价梯度异常
- 罕见错误共现
- 模板化痕迹明显

设计要求如下：

- 支持精确匹配、近似匹配、规则组合命中
- 支持规则编号、优先级、风险权重、版本号管理
- 支持输出命中证据和命中原因

### 8.5 围标特征规则引擎

围标特征规则引擎用于处理高确定性、可枚举的规则命中逻辑，重点负责以下分析：

- 联系人重合
- 电话、邮箱、地址重合
- 团队成员、资质、角色重合
- 报价规律异常
- 罕见错误共现

输出结果应包含：

- 命中规则标识
- 命中字段和值
- 命中证据锚点
- 风险权重或置信度

### 8.6 误报豁免引擎

误报豁免引擎是本场景的关键模块，用于识别“规则命中但合理”的内容，降低误报率。

至少应覆盖以下豁免场景：

- 法规、标准条文引用
- 招标模板固定表述
- 行业通用表达
- 低风险章节中的重复内容

建议处理策略如下：

- 建立法规引用库、模板白名单库、行业通用表述库
- 对引用段、标准条文、模板固定语进行降权
- 按章节类型设置不同权重
- 对同一命中规则结合章节权重、字段权重和共现情况调整风险级别

建议章节权重示例：

- 公司介绍：低权重
- 技术方案：高权重
- 服务承诺：高权重
- 风险控制：高权重
- 报价与关键参数：最高权重

### 8.7 风险融合层

风险融合层负责对多源评分、规则结果和豁免结果进行统一计算，不建议仅输出单一总分。

建议输出以下三个结果：

- 围标/陪标风险
- 整体审查风险
- 规则命中摘要

建议融合逻辑如下：

1. 优先处理硬规则命中，例如相同电话、相同邮箱、相同罕见错误、报价异常梯度。
2. 再融合章节权重、字段权重、共现规则和豁免结果。
3. 最终生成风险等级及解释项。

### 8.8 证据编排与报告 Agent

证据编排与报告 Agent 负责面向用户呈现结构化审查结果，不参与风险分值计算。

主要职责包括：

- 汇总高风险段落和表格
- 建立证据对照关系
- 生成审查摘要与复核建议
- 支持导出报告数据
- 响应“为什么判定为高风险”等追问

## 9. 数据对象设计

### 9.1 核心中间对象

建议定义以下核心对象：

- `Case`：审查任务实例
- `Document`：原始文档及元数据

- `Field`：结构化字段

## 10. 服务拆分建议

### 10.1 服务清单

建议将场景能力拆分为以下服务或工具层：

#### A. ParseTool

输入文档，输出结构化块、字段和锚点信息。

#### B. CollusionRuleTool

输出联系人重合、报价异常、团队重合等规则命中结果。

#### C. ExemptionTool

输出通用表达、法规引用、白名单模板等豁免结果。

#### D. ReportTool / ReviewTool

输出总体风险等级、高风险段落清单、判定理由和导出报告数据。

### 10.2 职责划分原则

- 场景 Agent 负责“调度”
- 工具服务负责“计算”
- LLM 负责“解释与生成”

该划分可显著降低核心判断链路的不可控性。

说明：

- `V1` 阶段不强依赖独立 `SimilarityTool`
- `V2` 阶段可新增语义召回与精排服务，以插件方式接入现有流程

## 11. LLM 使用边界

### 11.1 适合 LLM 的场景

- 将用户问题映射为场景任务
- 对证据进行自然语言解释
- 生成审查报告与复核建议
- 回答用户对结果的追问

### 11.2 不适合 LLM 直接承担的场景

- 全文两两相似度计算
- 最终风险分值计算
- 围标硬规则判断
- 违法违规结论认定

结论：LLM 应作为解释层与生成层能力，不应承担核心判定器角色。

## 12. 技术选型建议

### 12.1 Agent 编排框架

若继续采用 `LangChain4j`，建议优先使用以下能力：

- `AI Services`
- `Tools / Function Calling`
- `Structured Output`

建议原因如下：

- `AI Services` 适合封装稳定的场景调用接口
- `Tools` 适合调度解析、比对、规则等外部能力
- `Structured Output` 适合将模型输出直接映射为 Java 对象

不建议在本阶段将核心链路过度依赖实验性 agentic 能力，而应采用“固定工作流 + 工具调用 + 结构化输出”的实现方式。

### 12.2 V2 向量与语义能力扩展

`V1` 阶段暂不将向量与语义能力纳入主链路。

`V2` 阶段建议采用“两阶段检索 + 精排”的语义分析方案：

- 第一阶段：embedding 召回
- 第二阶段：pairwise rerank / CrossEncoder 复核

对于敏感场景，可优先考虑支持私有化部署或离线运行的模型方案，为后续医疗监管、内审场景复用提供基础。

## 13. 非功能要求

### 13.1 可解释性

- 高风险结果必须关联证据片段
- 证据必须可回溯到文档位置
- 风险结论必须可展示原因项

### 13.2 可维护性

- 规则可配置
- 阈值可调
- 章节权重可配置
- 白名单库和法规库可更新

### 13.3 可扩展性

- 支持从单次比对扩展到批量比对
- 支持从当前项目扩展到历史库检索
- 支持后续接入 OCR、多模态版面分析、关系图谱

### 13.4 安全性

- 支持敏感文档私有化处理
- 审查任务、结果和证据需具备权限控制
- 日志中避免记录原文敏感内容全文

## 14. V1 实施范围

`V1` 阶段建议聚焦“围标规则驱动的风险线索识别闭环”，范围如下：

- 支持 `docx` 文档解析
- 支持 `2~3` 份文档的单次审查
- 支持段落、表格、关键字段的结构化提取
- 支持联系人、团队、报价、地址、模板痕迹、罕见错误等规则命中
- 支持法规引用、模板白名单、行业通用表达的误报豁免
- 支持低、中、高三档风险等级输出
- 支持证据对照页和审查摘要导出

`V1` 阶段暂不纳入以下能力：

- 全文语义召回
- 全文语义精排
- 文本雷同/洗稿专用风险分数
- OCR、多模态版面恢复
- 大规模批量审查
- 跨项目关系图谱分析

## 15. V2 演进方向

`V2` 阶段在 `V1` 规则闭环稳定后，逐步增加全文语义评分能力，重点增强以下方向：

- 对全文段落和表格建立向量索引
- 在高价值章节中引入候选召回和语义精排
- 增加文本雷同、洗稿式改写、结构同源等风险识别能力
- 将语义分数作为新增特征接入风险融合层
- 在规则命中证据之外，补充语义证据对照能力

建议演进顺序如下：

1. 先在技术方案、服务承诺、风险控制等高价值章节试点语义比对。
2. 再扩展到全文范围和历史库候选检索。
3. 最后再考虑与关系图谱、批量审查做联合分析。

## 16. 当前版本结论

当前版本的技术设计以“规则驱动、证据可解释、流程稳定可落地”为核心。

结论如下：

- `V1` 先做围标规则命中主链路，优先证明业务价值和交付稳定性
- `V1` 不把全文语义分数放入核心判定链路，避免一开始工程复杂度过高
- `V2` 在现有结构化对象和证据体系上增量接入全文语义评分能力
- 整体架构保持不变，仅在分析服务层增加语义召回与精排模块

该方案能够在当前阶段帮助项目尽快形成可演示、可复核、可迭代的版本，同时为后续语义增强保留清晰扩展路径。
