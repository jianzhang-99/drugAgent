# 场景一标书审查测试计划与测试清单

> 文档版本：v1.0
> 更新时间：2026-03-18
> 适用范围：场景一 标书审查
> 文档目标：为标书审查场景的测试执行、缺陷收敛和验收交付提供专项测试计划与测试清单

---

## 1. 文档背景

本测试文档只围绕“场景一：标书审查”展开，不讨论整个 Drug-Agent 平台的全部能力。

结合当前设计文档，场景一的核心业务链路如下：

```text
任务接收
-> 文档解析
-> 结构化提取
-> 围标规则命中
-> 误报豁免
-> 风险融合
-> 证据编排
-> 报告生成
```

因此，本轮测试重点也应围绕这条链路组织，而不是泛化到所有场景。

---

## 2. 测试目标

本轮测试的目标如下：

- 验证标书审查主链路可完整跑通
- 验证 DOCX 上传、任务创建、文档解析和结构化提取稳定可用
- 验证围标规则命中结果具备可解释性
- 验证豁免、风险融合、证据编排结果合理
- 验证真实样本、合规样本、灰度样本上的输出表现
- 明确 AI 可辅助测试项与人工必测项，提升测试效率

### 2.1 测试通过标准

满足以下条件可判定场景一测试基本通过：

- `P0`、`P1` 缺陷清零
- 任务接收和文档解析主链路稳定可用
- 结构化提取结果满足下游字段要求
- 风险识别对高危样本有明显命中能力
- 合规样本不过度误报
- 风险结果能关联证据并支持人工复核
- 至少完成 1 轮回归测试和 1 轮样本验收测试

---

## 3. 测试范围

### 3.1 本轮重点范围

- 标书审查任务接收
- 文件上传与基础校验
- 文档解析
- 章节树提取
- 段落块、表格块提取
- 字段提取与标准化
- 围标规则命中
- 误报豁免
- 风险融合
- 证据编排
- 报告结果输出

### 3.2 本轮不作为重点

- 通用 Agent 全场景路由能力
- 合同预审场景
- 风险预警场景
- 分布式调度与高并发架构
- 生产级持久化恢复能力

---

## 4. 测试阶段安排

### 4.1 Phase 1：冒烟测试

目标：确认场景一具备进入详细测试的条件。

检查内容：

- 标书审查接口可访问
- 样本 DOCX 可上传
- 可成功创建任务
- 可解析指定文档
- 已有单元测试可执行

建议用时：0.5 到 1 天

### 4.2 Phase 2：功能测试

目标：验证主链路各阶段功能是否符合设计。

检查内容：

- 任务接收
- 文档解析
- 结构化提取
- 围标规则命中
- 豁免逻辑
- 风险融合
- 证据编排与报告结果

建议用时：2 到 3 天

### 4.3 Phase 3：异常与边界测试

目标：验证坏输入、边界输入和异常文件是否可控处理。

建议用时：1 到 2 天

### 4.4 Phase 4：真实样本与灰度样本测试

目标：用整理后的真实样本验证识别效果和误报控制。

建议用时：1 到 2 天

### 4.5 Phase 5：回归与验收测试

目标：修复缺陷后确认未引入回归，并形成最终验收结论。

建议用时：1 到 2 天

## 5. 样本分组建议

结合当前 `doc/场景一 标书审查/03_整理后样本数据`，建议按以下分组执行：

- 标准基准样本
  - 用于验证正常解析与低风险输出
- 高危样本对比组
  - 用于验证高风险规则命中能力
- 合法合规样本组
  - 用于验证误报控制能力
- 灰度与对抗演练测试组
  - 用于验证边界判断、解释能力和人工复核提示质量

## 7. AI 可辅助测试项与提示词

AI 在场景一测试阶段，最适合承担“初审”和“补充测试设计”工作，不适合替代人工做最终业务裁定。

### 7.1 样本问题与边界用例生成

用途：快速补充异常输入和边界文件测试思路。

推荐提示词：

```text
你是一名标书审查系统测试设计助手。请围绕以下场景生成测试用例：

场景：上传多份投标文件，系统进行解析、字段提取、围标规则命中、豁免、风险融合和报告输出。

请输出：
1. 正常用例
2. 异常用例
3. 边界用例

要求：
1. 每条用例包含：编号、输入条件、测试点、预期结果、优先级
2. 重点覆盖文件数量、文件格式、损坏文件、重复内容、字段缺失、章节异常、表格异常
```

### 7.2 结构化提取结果初审

用途：先让 AI 帮你看解析结果有没有明显问题，再由人工终审。

推荐提示词：

```text
你是一名标书结构化提取测试审查员。下面我会给你一份文档解析和字段提取结果，请从测试角度帮我初步检查。

请重点检查：
1. 章节树是否存在明显错层
2. 段落块和表格块是否存在明显缺失
3. 电话、邮箱、报价、人员字段是否提取正确
4. normalizedValue 和 normalizedKey 是否合理
5. blockId、anchor 是否支持回溯

请输出：
1. 总体判断
2. 可疑问题列表
3. 建议人工重点复核区域
```

### 7.3 围标规则命中结果初审

用途：辅助检查“命中了什么”和“为什么命中”是否基本一致。

推荐提示词：

```text
你是一名围标规则测试评审助手。下面我会给你系统输出的规则命中结果和部分原文证据，请帮我做测试初审。

请重点判断：
1. 命中规则与证据是否匹配
2. 是否存在证据不足但结论过强的情况
3. 是否存在应该命中但明显漏判的迹象
4. 哪些点需要人工复核

输出格式：
1. 初步判断
2. 可疑命中项
3. 可能漏判项
4. 人工复核建议
```

### 7.4 豁免与误报控制初审

用途：辅助判断系统是不是把模板化、法规化内容错误抬升了。

推荐提示词：

```text
你是一名误报控制测试助手。下面我会给你一份标书审查系统的风险命中结果、豁免结果和相关原文内容。

请从测试角度判断：
1. 哪些命中项看起来应当被豁免或降权
2. 哪些豁免项可能豁免过度
3. 是否存在合规样本被高风险误报的迹象

请输出：
1. 误报风险判断
2. 可能豁免不足的项
3. 可能豁免过度的项
4. 建议人工复核的样本和字段
```

### 7.5 回归测试包生成

用途：根据缺陷列表快速整理场景一专项回归包。

推荐提示词：

```text
请基于以下缺陷列表，为“标书审查场景”生成最小回归测试包。

要求：
1. 优先覆盖任务接收、文档解析、字段提取、规则命中、风险融合、报告输出
2. 每条回归项关联对应缺陷编号
3. 输出字段：回归项、覆盖模块、影响原因、执行方式、优先级
```

## 8. 人工必测项

以下测试项必须人工参与，不能完全交给 AI：

### 8.1 文件与解析链路

- 真实 DOCX 上传是否稳定
- 损坏文件和复杂表格文件是否触发异常
- 原文结构与章节树是否真实对齐
- 字段是否提取到了正确位置，而不是只是“看起来像对了”

### 8.2 业务判断终审

- 高危样本是否真的体现围标、陪标风险特征
- 合规样本是否存在误报
- 灰度样本是否被下了过强结论
- 风险等级是否符合业务直觉

### 8.3 证据与报告

- 风险结果是否能追溯到原文
- 证据链是否对人工复核有帮助
- 报告内容是否适合展示、汇报和答辩

### 8.4 演示验收

- 从上传样本到输出报告的全流程是否顺畅
- 页面展示和接口结果是否便于讲解
- 非开发人员是否能够理解输出内容

## 9. 建议测试执行顺序

建议按以下顺序执行：

1. 先跑现有单元测试和模块测试
2. 再做任务接收和文档解析冒烟测试
3. 再做结构化提取专项测试
4. 再做规则命中和豁免测试
5. 再做高危、合规、灰度样本测试
6. 再根据缺陷列表做回归测试
7. 最后做场景一专项验收演示

---

## 10. 缺陷分级建议

| 级别 | 定义 | 示例 |
|---|---|---|
| P0 | 阻断级 | 无法创建任务、无法解析文档、主链路中断 |
| P1 | 高优先级 | 关键字段大面积提取错误、风险结果明显不准、证据链缺失 |
| P2 | 中优先级 | 局部规则误判、部分章节解析偏差、报告可读性不足 |
| P3 | 低优先级 | 文案、展示格式、日志细节问题 |

## 12. 当前阶段建议

你现在最适合采用的方式是：

- 用这份文档做场景一专项测试总控
- 先围绕样本数据把主链路测透
- AI 负责初审和补充测试设计
