# 场景一标书审查数据结构对接文档

## 1. 文档目的

本文档只用于对齐“结构化提取”到“7.4 及后续模块”的数据交付格式。

开发范围从以下环节开始：

- `7.4` 围标规则命中
- `7.5` 误报豁免
- `7.6` 风险融合
- `7.7` 证据编排
- `7.8` 报告生成

因此，上游只需要按本文档提供结构化数据即可。

## 2. 需要交付的数据

上游统一输出一个 JSON 对象，包含 6 部分：

- `case` 本次任务是什么，用于标识这批数据属于哪一个业务审查任务，方便后续结果关联、状态流转和上下文记忆
- `documents` 本次任务涉及的文档清单，用于说明有哪些输入文档参与本次审查
- `compareScopes` 哪些文档需要进入比对范围，用于告诉后续引擎应该在什么边界内做比较
- `blocks` 从原文切割出来的内容块，是后续规则命中、证据展示、原文跳转的基础载体
- `fields` 从原文里提取出来的结构化字段，是规则命中、风险融合真正消费的核心输入
- `extractionMeta` 这批结构化数据的技术说明，用于记录它是按什么 schema、什么解析器版本生成出来的

结构如下：

```json
{
  "case": {},
  "documents": [],
  "compareScopes": [],
  "blocks": [],
  "fields": [],
  "extractionMeta": {}
}
```

## 3. 必填字段

### 3.1 case

`case` 用于描述“这批结构化数据属于哪一个业务任务”。它不是文档内容本身，而是任务级别的身份信息。后续的规则命中、风险结果、报告产物，通常都会挂在同一个 `caseId` 下，所以这里的字段主要承担“任务归属”和“业务场景标识”的作用。

字段说明：

- `caseId`：审查任务唯一 ID，用来标识这批数据属于哪一次审查任务。后续无论是规则命中、风险分数还是报告结果，都需要通过它挂回具体任务。
- `scene`：业务场景类型，用来说明这是哪一类业务场景。这里建议固定为 `tender_review`，表示标书审查。之所以单独设计这个字段，是为了方便未来扩展到合同审查、资质审查等其他场景。

| 字段 | 类型 | 说明 |
|------|------|------|
| `caseId` | string | 审查任务 ID |
| `scene` | string | 固定建议为 `tender_review` |

### 3.2 documents

`documents` 用于描述“本次任务中有哪些输入文档”。因为后续所有 `blocks`、`fields`、比对范围、证据展示，最终都要落回具体文档，所以文档清单必须单独建模，而不能只在后面的明细里零散出现。

字段说明：

- `documentId`：文档唯一 ID，是后续 `blocks`、`fields` 关联文档时使用的主键。单独使用 ID，而不是直接用文件名做关联，是为了避免重名、改名等问题。
- `documentName`：文档展示名称，主要用于前端展示、报告展示和人工核查。它更偏向“给人看”的信息。
- `fileType`：文档类型，例如 `docx`、`pdf`。这个字段的作用是帮助后续模块判断文档来源和解析方式，也方便前端做预览适配。

| 字段 | 类型 | 说明 |
|------|------|------|
| `documentId` | string | 文档 ID |
| `documentName` | string | 文档名称 |
| `fileType` | string | 文件类型 |

### 3.3 compareScopes

`compareScopes` 用于描述“哪些文档应该放在一起比较”。并不是所有输入文档都天然需要参与同一轮比对，所以比对范围本身需要被显式表达出来。这样后续规则引擎才能知道比较边界，避免不必要的全量比较。

字段说明

- `scopeId`：比对范围唯一 ID，用于标识一个独立的比较分组。
- `scopeType`：比对范围类型，用于说明这一组文档按什么方式比较。当前建议值为 `full_bid_compare`，表示整份投标文件范围内比较。
- `documentIds`：参与该比对范围的文档 ID 列表，用于明确告诉后续引擎，哪些文档应当进入同一组比较。

| 字段 | 类型 | 说明 |
|------|------|------|
| `scopeId` | string | 比对范围 ID |
| `scopeType` | string | 建议值：`full_bid_compare` |
| `documentIds` | string[] | 参与比对的文档 ID 列表 |

### 3.4 blocks

`blocks` 用于原文定位、证据展示、前端跳转。

这里的 `blocks` 可以理解为“原文证据块”。字段提取出来以后，系统不仅要知道“提取了什么”，还要知道“这些内容原来出现在文档的什么位置、以什么上下文出现”。因此 `blocks` 这一层是后续证据编排和原文追溯的基础。

字段说明

- `blockId`：内容块唯一 ID，用于让 `fields` 能够回溯到来源原文块。
- `documentId`：所属文档 ID，用于说明该内容块来自哪份文档。
- `blockType`：内容块类型，例如 `paragraph`、`table`、`field`。设计这个字段，是为了让后续系统知道这段内容原来以什么形式存在后续可能有不同规则处理。
- `chapterPath`：章节路径，用于补充块所在的语义上下文。相同内容出现在不同章节时，业务意义可能完全不同。
- `content`：规范化文本，用于机器处理和规则比较。通常会做基础清洗，例如去除多余空白、统一格式。
- `rawContent`：原始文本，用于证据展示、人工核查和问题排查。它保留更接近原文的内容。
- `anchor`：锚点信息，用于告诉前端和后续模块这段内容在原文中的具体位置，是做跳转、高亮、证据定位的关键字段。

| 字段 | 类型 | 说明 |
|------|------|------|
| `blockId` | string | 内容块 ID |
| `documentId` | string | 所属文档 ID |
| `blockType` | string | `paragraph/table/field` |
| `chapterPath` | string | 章节路径 |
| `content` | string | 规范化文本 |
| `rawContent` | string | 原始文本 |
| `anchor` | object | 锚点信息 |

`anchor` 至少提供一种稳定定位信息，核心目的是让后续系统能够回到原文找到证据，并且方便前端处理，建议包含：

- `pageNo`
- `sectionNo`
- `paragraphNo`
- `tableNo`

### 3.5 fields

`fields` 是后续规则命中和风险融合的核心输入。

`fields` 是从原文中提取出来的结构化字段，可以理解为“引擎真正消费的数据”。因为规则命中、误报豁免、风险融合更适合在标准化字段上计算，而不是直接在大段原文上处理，所以这里需要把可计算的信息单独抽出来。

字段说明

- `fieldId`：字段唯一 ID，用于唯一标识一个提取结果，方便后续命中结果、风险结果精确引用。
- `documentId`：所属文档 ID，用于说明该字段来自哪份文档，也是跨文档比较时的重要关联键。
- `blockId`：来源块 ID，用于把结构化字段和原文证据块关联起来，确保字段可以回溯到原文。
- `fieldType`：字段类型，用于告诉后续引擎这是什么类别的数据，例如电话、邮箱、地址、人员、报价等。
- `fieldName`：字段名称，偏向展示用途，便于前端和报告直接向用户展示。
- `fieldValue`：原始字段值，保留从原文中抽出来的原始表达，便于人工核查。
- `normalizedValue`：标准化字段值，用于统一不同表达形式，方便后续做比较和计算。
- `normalizedKey`：归一化聚合键，用于后续聚合、命中和相同值判断。建议带上类型前缀，例如 `phone:`、`email:`，这样可以减少歧义。
- `chapterPath`：字段所在章节路径，用于给字段补充章节上下文，避免后续模块每次都必须先回查 `blocks`。
- `anchor`：字段来源锚点，用于字段级别的证据定位和前端跳转。
- `confidence`：提取置信度，用于描述该字段抽取结果的可靠程度，便于后续做降权、排错或人工复核。

| 字段 | 类型 | 说明 |
|------|------|------|
| `fieldId` | string | 字段 ID |
| `documentId` | string | 所属文档 ID |
| `blockId` | string | 来源块 ID |
| `fieldType` | string | 字段类型 |
| `fieldName` | string | 字段名称 |
| `fieldValue` | string | 原始字段值 |
| `normalizedValue` | string | 标准化字段值 |
| `normalizedKey` | string | 归一化聚合键 |
| `chapterPath` | string | 章节路径 |
| `anchor` | object | 来源锚点 |
| `confidence` | number | 提取置信度 |
### 3.6 extractionMeta

`extractionMeta` 用于描述“这批结构化结果是怎么生成出来的”。它不是业务任务信息，而是这份 JSON 的技术说明。后续当解析结果异常、版本升级、字段结果发生变化时，可以通过这里的信息快速定位原因。

字段说明

- `schemaVersion`：当前结构化数据所遵循的 schema 版本，用于保证上下游对字段含义的理解一致。
- `parserVersion`：生成这批结构化数据所使用的解析器版本，用于版本追踪和问题排查。
- `parseSuccess`：解析是否成功，用于告诉下游这批数据是不是在正常解析完成的前提下生成的。

| 字段 | 类型 | 说明 |
|------|------|------|
| `schemaVersion` | string | schema 版本 |
| `parserVersion` | string | 解析器版本 |
| `parseSuccess` | boolean | 是否解析成功 |

## 4. 关键约束

- `blocks` 和 `fields` 必须能通过 `blockId` 关联，这样字段才能回溯到原文证据块
- `fields.normalizedValue` 和 `fields.normalizedKey` 必须由上游完成，避免不同下游模块各自重复做标准化，导致规则不一致
- `anchor` 不能省略，否则后续无法做证据定位、原文跳转和报告引用
- 关键字段缺失时，可以给空值，但不要直接省略字段，这样有利于保持接口结构稳定
- 所有 ID 必须唯一，避免引用混乱和数据串联错误

`normalizedKey` 建议格式：

- 电话：`phone:13812345678`
- 邮箱：`email:zhangsan@test.com`
- 人员：`person:张三`
- 地址：`address:北京市朝阳区xxx`
- 资质：`qualification:药品经营许可证`
- 报价总额：`quote_total:1250000.00`
